# サービスレイヤー

このドキュメントでは、ローカルメールアプリケーションのサービスレイヤーを説明します。

## 概要

サービスレイヤーは`srv/`フォルダにあり、ODataサービス定義とビジネスロジックを含んでいます。

## ファイル構造

```
srv/
├── cat-service.cds     # ODataサービス定義
└── cat-service.js      # サービス実装（オプション）
```

## サービス定義

### OData V4サービス

サービスは、`db/schema.cds`で定義された`Emails`エンティティを公開するOData V4エンドポイントを提供します：

```cds
using my.emailapp2 as my from '../db/schema';

@path : '/odata/v4/email'
service CatalogService {
    @odata.draft.enabled
    entity Emails as projection on my.Emails;
}
```

| 要素 | 説明 |
|------|------|
| `@path` | サービスエンドポイントURL - `/odata/v4/email` |
| `CatalogService` | サービス名 |
| `@odata.draft.enabled` | エンティティにドラフト機能を有効化 |
| `entity Emails` | `Emails`テーブルの投影 |

## サービスエンドポイント

デプロイされると、次のエンドポイントが利用可能になります：

- **サービスメタデータ**: `http://localhost:4004/odata/v4/email/$metadata`
- **エンティティセット**: `http://localhost:4004/odata/v4/email/Emails`
- **個別エンティティ**: `http://localhost:4004/odata/v4/email/Emails(<ID>)`

## ドラフト対応

`@odata.draft.enabled`注釈により、次の機能が有効になります：

1. **ドラフト作成**: ユーザーは未保存のドラフトを作成できます
2. **ドラフト編集**: 既存のエンティティをドラフトとして編集できます
3. **保存/破棄**: ドラフトを保存してアクティブにするか、変更を破棄できます

## ユーザーフィルタリング

アプリケーションはユーザースコープデータアクセスを実装しています。これは主に2つの方法で実現されます：

### 1. データベースレベル（所有者フィールド）

`Emails`エンティティには、各レコードを作成したユーザーを追跡する`owner`フィールドがあります。

### 2. サービスレベルフィルタリング

サービス実装は、リクエストをユーザーのレコードのみに自動的にフィルタリングできます。これは通常、カスタムJavaScriptハンドラ（`cat-service.js`）で実装されます。

## 技術的詳細

### CDS投影
サービスはデータベースエンティティの**投影**を使用します。これにより：
- すべてのデータベースフィールドが公開されます
- サービス固有の追加フィールドまたは計算フィールドを追加できます
- 名前空間の衝突を防ぎます

### OData注釈
サービス定義には、以下のようなさまざまなUI注釈を含めることができます：
- `@readonly` - エンティティに対する作成/更新/削除を禁止
- `@insertonly` - 作成のみ許可
- `@updateonly` - 更新のみ許可
